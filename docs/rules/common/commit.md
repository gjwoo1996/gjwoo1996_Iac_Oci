# 커밋 규칙

**규칙 ID:** `common/commit`  
**적용 대상:** 이 프로젝트에서 작업하는 모든 AI  
**적용 범위:** 모든 git 커밋

---

## 원칙

### 1. 커밋 분리 원칙

변경사항을 커밋하기 전에 반드시 내용을 검토하여, **분리 가능한 변경사항은 별도의 커밋으로 나눈다.**

하나의 커밋에는 하나의 목적만 담는다.

**분리 기준:**

| 상황 | 처리 방법 |
|------|-----------|
| 기능 추가 + 버그 수정이 동시에 발생한 경우 | 별도 커밋으로 분리 |
| 여러 파일에 걸친 독립적인 변경사항 | 관련된 것끼리 묶어 분리 |
| 리팩토링 + 기능 변경이 함께 있는 경우 | 리팩토링 먼저, 기능 변경 별도 커밋 |
| 문서 수정 + 코드 수정이 함께 있는 경우 | 별도 커밋으로 분리 |
| 규칙·설정 변경 + 실제 작업이 함께 있는 경우 | 별도 커밋으로 분리 |

**예외 — 하나의 커밋으로 묶어도 되는 경우:**
- 변경사항이 서로 강하게 연결되어 있어 분리하면 코드가 동작하지 않는 경우
- 한 줄짜리 오타 수정처럼 분리의 의미가 없는 경우

---

### 2. 커밋 메시지 작성 원칙

커밋 메시지는 **무엇을, 왜 변경했는지** 다른 사람이 코드를 보지 않고도 이해할 수 있도록 작성한다.

#### 형식

```
<타입>: <제목>

<본문>
```

#### 타입 목록

| 타입 | 사용 상황 |
|------|-----------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 추가·수정 |
| `refactor` | 동작 변경 없이 코드 구조 개선 |
| `chore` | 빌드·설정·도구 변경 |
| `infra` | 인프라 코드 변경 (IaC) |
| `test` | 테스트 추가·수정 |
| `style` | 포맷·들여쓰기 등 코드 스타일만 변경 |

#### 제목 작성 규칙

- 50자 이내로 간결하게 작성한다.
- 동사 원형으로 끝맺는다. ("추가", "수정", "삭제", "개선", "제거")
- 끝에 마침표를 붙이지 않는다.

#### 본문 작성 규칙

- 제목만으로 의도가 충분히 전달되지 않을 때 작성한다.
- **무엇을** 바꿨는지보다 **왜** 바꿨는지를 중심으로 작성한다.
- 변경 전/후의 동작 차이가 있다면 명시한다.
- 항목이 여러 개면 `-` 또는 `·` 로 목록 형태로 작성한다.

#### 작성 예시

**좋은 예:**
```
feat: devcontainer에 OCI CLI 설치 추가

- OCI 리소스 조회 및 관리를 컨테이너 내부에서 바로 수행하기 위해 추가
- ~/.oci를 read-only로 마운트하여 호스트 인증 정보를 재사용
```

```
docs: 언어 사용 규칙 문서 추가

모든 AI가 답변을 한국어로 작성하도록 공통 규칙 정의
성능상 다른 언어가 필요한 경우 사용자 고지 절차 포함
```

```
fix: OCI CLI 인증 경로 오류 수정
```

**나쁜 예:**
```
update files         ← 무엇을 했는지 알 수 없음
fix                  ← 무엇을 고쳤는지 알 수 없음
작업 완료            ← 내용 없음
여러 가지 수정       ← 분리해야 할 내용을 하나로 묶음
```

## AI를 위한 커밋 전 체크리스트

커밋 전 반드시 아래를 확인한다:

- [ ] `git diff`로 변경사항 전체를 확인했는가?
- [ ] 하나의 커밋에 서로 다른 목적의 변경사항이 섞여 있지는 않은가?
- [ ] 분리할 수 있는 변경사항은 별도 커밋으로 나눴는가?
- [ ] 커밋 메시지 제목만 보고도 변경 내용을 파악할 수 있는가?
- [ ] 변경 이유가 불명확하다면 본문에 이유를 작성했는가?
